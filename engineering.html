Engineering Philosophy & Architecture

This page represents how I think as an engineer — from architecture decisions to multiplayer synchronization, optimization, and scalable systems design in Unity.

1. Clean Architecture in Unity

I treat Unity projects as software systems, not just game scenes. My architecture focuses on maintainability, scalability, and testability.

  • Separation of concerns between UI, gameplay, networking, and data layers
  • Use of MVP pattern for UI-heavy systems
  • ScriptableObjects for configuration, stats, and reusable data
  • Singletons only where lifecycle guarantees are required
SOLID MVP OOP Clean Architecture

2. Multiplayer Systems (Photon PUN)

Multiplayer engineering is about state consistency, latency handling, and player experience under unreliable networks.

Core systems I implemented:

  • Room-based matchmaking with dynamic player counts (1–4 players)
  • Player ready-state synchronization before game start
  • Master-client authoritative flow where applicable
  • Scene synchronization using AutomaticallySyncScene

I prefer Photon Custom Properties and RaiseEvent for deterministic state synchronization instead of overusing RPCs.

Photon PUN Room Properties RaiseEvent Sync Scene

3. AI Architecture (State & Strategy)

Enemy behavior is implemented using a modular AI system rather than monolithic Update logic.

  • State Machine: Idle, Move, Attack, Dead
  • Strategy Pattern for movement & targeting behaviors
  • ScriptableObject-based enemy stats and configurations
  • Object pooling for enemies and bullets

This allows new enemy types to be created without modifying core logic.

State Pattern Strategy Pattern AI Design

4. Performance & Optimization Mindset

Performance optimization is considered from day one, not as a post-production fix.

  • Layer Collision Matrix instead of manual collision checks
  • Network data size reduction and compact serialization
  • Texture offset scrolling instead of repeated environment spawning
  • Pooling to avoid runtime allocations

I actively profile CPU, GC, memory, and network traffic during development.

Optimization Profiler GC Control

5. Backend & Data Flow

I design backend systems to support real production needs such as persistence, authentication, and scalability.

  • Node.js + Express REST APIs
  • MongoDB for structured game data storage
  • Firebase for authentication and cloud services
  • Secure UnityWebRequest integration

Game data is serialized carefully to maintain compatibility between client and server versions.

Node.js MongoDB Firebase REST APIs

6. How I Think as an Engineer

  • Build systems that are easy to extend, not just easy to finish
  • Prefer clarity and correctness over clever shortcuts
  • Optimize based on measurements, not assumptions
  • Write code juniors can understand and seniors can respect

My long-term goal is to grow into a game systems architect who designs multiplayer games that scale technically and creatively.
