Multiplayer Card Game

A real-time multiplayer card game built with Unity and Socket.IO, focusing on synchronization, turn-based logic, room management, and scalable client-server communication.

This project is a **real-time multiplayer card game** designed to demonstrate my expertise in **socket programming, event-driven networking, turn-based systems, and clean multiplayer architecture**.

The game follows a **client-server model**, where the server acts as the authoritative source of truth to ensure fairness, consistency, and cheat prevention.

  • Turn-based card mechanics with server-side validation
  • Authoritative server logic for game state control
  • Player join / leave handling without breaking game flow
  • Consistent card state synchronization across all clients
  • Reconnect-safe gameplay with state rehydration

üõ† Technical Architecture

  • MVP / Clean Architecture for UI and game logic separation
  • Server-authoritative game state management
  • Socket.IO event-based communication for gameplay actions
  • State Machines for game phases (Waiting ‚Üí Playing ‚Üí Result)
  • Data-driven card definitions using ScriptableObjects
  • Optimized network payloads to minimize latency and bandwidth

üåê Multiplayer & Networking Logic

  • Room-based matchmaking using Socket.IO namespaces / rooms
  • Server-controlled turn order and action validation
  • Event-driven synchronization instead of continuous polling
  • Custom socket events for turn updates and card actions
  • Graceful disconnect handling with server-side recovery

Unity (C#) Socket.IO Node.js ScriptableObjects State Machine MVP Pattern Git Android / Windows

üöß Challenges & Solutions

  • Challenge: Keeping all players synchronized in real time
    Solution: Used Socket.IO event-driven updates with server-authoritative validation.
  • Challenge: Handling disconnects during active turns
    Solution: Maintained game state on the server and re-sent state on reconnect.
  • Challenge: Preventing invalid or duplicate actions
    Solution: All actions are validated server-side before being broadcast.
